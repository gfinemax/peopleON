# 개발단계 용어집 (1페이지)

개발단계에서는 아래 용어만 먼저 통일해서 사용합니다.

| 용어 | 쉬운 의미 | 주로 보는 화면 | 비고 |
|---|---|---|---|
| 인물(Person) | 조합원/권리증 보유자/관계인 포함 대상자 | `/members` | 관리의 시작점 |
| 케이스(Settlement Case) | 한 인물의 환불 계산 단위 | `/settlements` | 상태(`작성중/검토중/승인/지급완료/반려`) |
| 정산예정 | 환불 기준으로 계산된 총액 | `/settlements`, `/members` | `final_refund` 합 |
| 지급 | 실제 지급된 금액 | `/settlements`, `/payments` | `refund_payments.paid` 합 |
| 잔여 | 정산예정 - 지급 | `/settlements`, `/members` | 0보다 크면 미지급 |
| 출자금 | 조합/신탁 계좌로 입금된 금액 | `/finance` | 기본적으로 자본성 |
| 차입성 금액 | 조합 외 계좌로 들어온 금액(계약 근거 필요) | `/finance` | 개발단계는 차입성으로 임시 분류 |
| 보류(suspense) | 성격을 아직 판단 못한 금액 | `/finance` | 증빙 확보 후 재분류 |
| 권리증 기준 환불 | 권리증 건당 기준 환불금(예: 3,000만원) | `/settlements` | 정책값 기반 |
| 프리미엄 인정 | 기준 환불 외 추가 인정액 | `/settlements` | 별도 라인 관리 |
| 매몰비용/손실 | 사업 과정 손실 반영 금액 | `/settlements` | 최종 환불 계산에 영향 |
| 재분류 | 금액 성격을 사유와 함께 변경 | `/finance` | 감사로그 필수 |
| QA 실행 | 데이터 정합성 자동 점검 | `/settlements` | warn/fail 추적 |
| 상태 동기화 | 잔여 기준으로 케이스 상태 자동 보정 | `/settlements` | 지급완료 불일치 교정 |
| 감사로그(Audit) | 누가/언제/무엇/왜 변경했는지 기록 | DB(`audit_logs`) | 분쟁 대응 핵심 |

## 개발단계 원칙

1. 용어는 쉬운 업무어 우선 사용
2. 회계 확정이 아닌 항목은 `임시 분류`로 처리
3. 임시 분류는 반드시 재분류 가능하도록 설계
4. 모든 변경은 감사로그로 남김
